## 1. 背景与核心结论

### 目标
用户希望**自动化抓取自己在微信里关注的多个视频号的作品列表**，一次性把「所有关注博主的视频清单」采集到本地，而不是现在这样一个博主一个博主手动点「同步作品」。

### 核心结论（先读这一条，它决定了整个方案的形态）

**不需要 OS 级别的"自动点鼠标操作微信"。** 现有注入体系已经把视频号 App 内部的协议级 API 暴露给了注入脚本，采集所需的三块能力**两块已具备、一块已暴露但未使用**：

| 能力 | API | 现状 |
|---|---|---|
| 枚举「我关注的视频号」 | `WXU.API4.finderGetFollowList(payload)` | ✅ **已被 bundle hook 暴露**（`utils.js:712` 检测、`utils.js:745` getter），但**全项目没有任何地方调用它** ← 这是本方案要补的唯一新 API |
| 翻页拉取单个作者的作品 | `WXU.API.finderUserPage({username, finderUsername, lastBuffer, …})` | ✅ 已在用（`profile.js:146` `__wx_load_one_page`） |
| 把作品落盘（含解密） | `POST /__wx_channels_api/sync-feed`（仅元数据）/ `/__wx_channels_api/download`（下视频文件） | ✅ 已实现（`mitm_proxy.py:561` `save_synced_feeds`、`channels.py:578`） |

因此**自动化采集 = 一段注入到 webview 里的"编排脚本"**：

```
finderGetFollowList  →  得到关注的作者 [{username, nickname, headUrl}, ...]
        │
        └─ for each author:
               finderUserPage(lastBuffer 翻页) → 拿到原始 feeds（含 decodeKey）
               POST /__wx_channels_api/sync-feed { username, feeds }   ← 复用现有后端
               （带节流，模拟人类节奏）
```

整个过程**在 WeChat 内置浏览器的页面上下文里跑**，请求自动携带登录态，**无需任何点击/像素自动化**。

### 唯一需要「碰微信」的地方：bootstrap
注入脚本只有在 webview 加载过一次视频号页面后才会生效（bundle 被 hook、`WXU.API*` 才被填充、登录态才在 webview 里）。本方案 v1 采用**手动 bootstrap**：

> 用户在微信里点开任意一个视频号页面（或工具内置浏览器打开 `channels.weixin.qq.com`），注入脚本会自动在页面上出现一个「**一键采集全部关注**」按钮，点一下即开始。

这意味着 **v1 几乎不需要改后端**——编排逻辑全在注入脚本里，结果走现成的 `sync-feed`，落到现成的 `channels_parsed_feeds.json` + `channels_favorites.json`，工具箱原有的视频号页面直接就能看到采集结果。

## 2. 范围（已与用户确认）

| 维度 | 决策 | 含义 |
|---|---|---|
| **默认行为** | **仅同步元数据** | 采集时只调 `sync-feed` 把作品清单（标题/封面/CDN URL/`decodeKey`/createtime）写进 `channels_parsed_feeds.json`，**不下载视频文件**。下载留给用户事后在作者作品列表里按需点。体积小、最快、风控风险最低。 |
| **启动方式** | **手动打开一次即可** | 用户手动在微信打开视频号页 → 注入脚本自动出现「一键采集全部关注」按钮。不做 Playwright 自动开页，不做 OS 自动点微信。 |

**v1 明确不做（避免过度设计）**：自动下载视频文件、工具箱 UI 远程触发采集、OS 级微信自动化、定时自动采集。这些列在 §7「未来扩展」，**不在本次实现范围**。

## 3. 关键约束（实现前必须理解）

1. **复用后端契约，不改后端下载/同步逻辑。** `sync-feed` 期望**原始 feed 结构**（读 `feed.objectDesc.media[0].decodeKey`、`feed.objectDesc.mediaType===4`），`profile.js:536` 的同步路径已经验证过这条链路可用——**直接照抄它的 `rawObjects.filter(mediaType===4)` → sync-feed 写法**，不要自己造新格式。

2. **`finderGetFollowList` 的入参/出参 schema 未知**，是本方案唯一的"未知数"。必须由 Recon 工作流先在真机/控制台里探明（见工作流 A）。**不要凭空臆造字段名**。参考实现里若有线索（`wx_channels_download_temp/`）优先采信，否则运行时实测。

3. **节流是硬性要求，不是可选项。** 连续枚举几十个作者、每个翻几页，是明显的非人类行为，极易触发视频号风控（轻则接口 4xx/限流，重则封号风险）。项目里已有现成的对数正态延迟工具 `lognormal_sleep`（小红书模块在用，见 `backend/xiaohongshu.py`）——注入脚本侧需要一个等价的 JS 版随机延迟。**作者之间、翻页之间都要等。**

4. **必须可取消、可断点续采。** 采集是长耗时任务，要能中途停；已采过的作者要能跳过（依据 `channels_parsed_feeds.json` 里是否已有该 username 的数据）。

5. **正则/注入顺序的坑见移植计划文档**——本方案新增的 `automation.js` 只是**又一个页面脚本**，挂在依赖链最后（`components.js` 之后），不碰 bundle hook、不碰注入顺序的前置部分。

### 工作流 B — 注入编排器 `automation.js`

**依赖**：工作流 A 的 schema 结论。

**目标**：新增 `injection_scripts/src/automation.js`，在视频号页面插入「一键采集全部关注」按钮，点击后跑完整采集循环。

**实现要点**（保持简单，复用现有件）：

1. **复用现有 UI/工具**，不要重造：
   - 进度提示直接用 `profile.js` 里的 `showProgressCard/hideProgressCard` 模式（可把它们抽到 `components.js` 或在 `automation.js` 内联同款）。
   - 同步出口完全照抄 `profile.js:500-548` 的 `rawObjects.filter(obj => obj.objectDesc?.mediaType===4)` → `POST /__wx_channels_api/sync-feed {username, feeds}`。
   - 翻页照抄 `__wx_load_one_page`（`profile.js:146`）的 payload 结构和 `hasMore = !!(lastBuffer && rawObjects.length >= 15)` 判定。

2. **JS 版节流**（新增，必须有）：
   ```js
   // 对数正态随机延迟，模拟人类（对齐后端 lognormal_sleep 思路）
   function jitterSleep(baseMs) {
     const ms = baseMs * Math.exp((Math.random() - 0.5) * 0.8); // 粗略对数正态
     return new Promise(r => setTimeout(r, ms));
   }
   // 翻页之间 ~1.5s，作者之间 ~3s（可调，宁慢勿快）
   ```

3. **主循环 `harvestAllFollows(opts)`**：
   ```
   authors = []  ; cursor = ""
   do {                                   // A 给的翻页范式
     page = await WXU.API4.finderGetFollowList({cursor})
     authors.push(...page.items.map(normalizeFollowItem))
     cursor = page.nextCursor ; await jitterSleep(1500)
   } while (page.hasMore && !cancelled)

   for (const author of authors) {        // 作者之间 jitterSleep(3000)
     if (skipAlreadySynced && alreadyHas(author.username)) continue;  // 断点续采
     // 把作者先加进收藏（复用 sync-feed 的副作用，或单独 POST /api/channels/favorites）
     let marker = "" ; let hasMore = true;
     while (hasMore && !cancelled) {
       const r = await WXU.API.finderUserPage({
         username: author.username, finderUsername: my_username,
         lastBuffer: marker, needFansCount: 0, objectId: "0"
       });
       if (r.errCode !== 0) { logError(r.errMsg); break; }  // 限流则退避，见下
       const raw = r.data.object || [];
       const videos = raw.filter(o => o.objectDesc && o.objectDesc.mediaType === 4);
       if (videos.length) await WXU.request({method:"POST", url:"/__wx_channels_api/sync-feed",
                                              body:{username: author.username, feeds: videos}});
       marker = r.data.lastBuffer || ""; hasMore = !!(marker && raw.length >= 15);
       await jitterSleep(1500);
     }
     updateProgressCard(已采 N/总 M 作者, 本作者 X 条);
   }
   ```

4. **限流退避**：`finderUserPage`/`finderGetFollowList` 返回非 0 errCode（典型限流）时，**指数退避**（等 5s→15s→45s），连续失败 3 次则**暂停并提示用户**，不要硬刚。

5. **可取消**：按钮点击后变「停止采集」，置 `cancelled=true`；循环每步检查。

6. **断点续采**：`alreadyHas(username)` 需要知道 `channels_parsed_feeds.json` 是否已有该 username。两种实现，二选一：
   - (a) 页面加载时拉一次 `GET /api/channels/favorites` 或新增一个轻量 `GET /api/channels/synced-usernames`（返回已同步的 username 集合）——**推荐**，准确。
   - (b) 纯前端：本次会话内记忆已采过的，不跨会话。简单但重启后重采。
   v1 用 (a)，需要工作流 C 加一个只读端点。

**约束**：`automation.js` 只读 `WXU.*`，不碰 bundle hook、不碰 `lib/`。文件挂在注入链最后。

**验证标准**（真机）：
1. 视频号页面出现「一键采集全部关注」按钮，不白屏。
2. 点击后 ProgressCard 显示「正在枚举关注… / 正在采集作者 X (N/M)」。
3. 跑完后 `data/channels_parsed_feeds.json` 里**新增多个 username 的条目**，每条 `decode_key` 非空；`channels_favorites.json` 里出现这些作者。
4. 全程接口无大面积 4xx；中途点「停止」能立即停。
5. 工具箱「微信视频号」页面刷新后能看到这些作者及其作品。

## 6. 风险与合规（必须向用户明示，写进 UI）

1. **这是对微信视频号平台的批量抓取**。即便只抓自己关注的、只抓元数据，**高频枚举仍可能触发风控**，存在账号被限制功能甚至封禁的风险。节流（§3.3）是为降低概率，**不能消除风险**。
2. **仅用于个人备份/学习用途**；采集到的内容版权属原作者，不得二次传播或商用。
3. **请求量自我设限**：v1 建议默认对单作者设上限（如最多翻 N 页 / 最多 X 条），并在 UI 暴露；用户主动调高时给风险提示。这是"无声截断要说出来"原则——若设了上限，进度卡要显示"已达上限，未采完"。
4. CA 证书 / 系统代理 / 缓存清理等敏感操作沿用现有实现，**本方案不新增任何系统级权限操作**。

## 7. 未来扩展（明确不在 v1 范围，仅备忘）

| 扩展 | 摘要 | 主要难点 |
|---|---|---|
| 同步即下载 | 采集时顺带 `POST /__wx_channels_api/download` 下视频文件 | 体积/耗时/风控放大；建议下载与采集解耦，事后批量下 |

| OS 级微信自动化 | pywinauto/AppleScript 点微信原生界面 | 跨版本极脆、维护成本高，仅作最末兜底 |

## 8. Recon 结论（工作流 A 已完成 — 真机验证通过）

> ✅ 已于真机跑通。`WXU.API4.finderGetFollowList` 确实暴露且可用。原始 dump 见 `data/follow_list_recon.json`（侦察完已删，结构如下）。

### 8.1 入参
空对象即可拉第一页；翻页传上一页返回的 `lastBuffer`：
```js
await WXU.API4.finderGetFollowList({ lastBuffer: "" });   // 第一页
await WXU.API4.finderGetFollowList({ lastBuffer: r.data.lastBuffer }); // 下一页
```

### 8.2 出参结构（`r` = 调用返回）
```
r.errCode        // 顶层：0=成功；-70002 = JSAPI_TRANSFER_FAILED（见 8.4 重试）
r.errMsg         // "ok" / "JSAPI_TRANSFER_FAILED"
r.data.BaseResponse.Ret        // 0
r.data.contactList[]           // 关注的作者数组（本页）
   ├─ username    // "v2_xxxxxxxx@finder"  ← 喂给 finderUserPage 的 username
   ├─ nickname    // 昵称
   ├─ headUrl     // 头像
   ├─ signature / followTime / authInfo / liveStatus / extInfo ...  // 其余元数据，采集用不到
r.data.lastBuffer     // 翻页游标，传给下一次调用
r.data.continueFlag   // 0 = 没有更多了；非 0 = 还有下一页
r.data.followCount    // 关注总数（可用于进度显示）
```

### 8.3 归一化 + 翻页范式（B 直接用）
```js
function normalizeFollowItem(c) {
  return { username: c.username, nickname: c.nickname, headUrl: c.headUrl };
}

async function fetchAllFollows() {
  const authors = [];
  let lastBuffer = "";
  while (true) {
    const r = await callWithRetry(() => WXU.API4.finderGetFollowList({ lastBuffer }));
    if (!r || r.errCode !== 0) break;
    const list = (r.data && r.data.contactList) || [];
    list.forEach(c => authors.push(normalizeFollowItem(c)));
    lastBuffer = (r.data && r.data.lastBuffer) || "";
    if (!r.data || r.data.continueFlag === 0 || !lastBuffer || list.length === 0) break; // 没有更多
    await jitterSleep(1500); // 翻页节流
  }
  return authors; // 可按 username 去重
}
```
> 实测：`followCount: 8` 的账号一页返回全部 8 条、`continueFlag: 0`。翻页逻辑以 `continueFlag` 为主、`lastBuffer`/空列表为兜底。

### 8.4 ⚠️ 必须处理：API 未就绪的瞬时拒绝
真机首次页面加载时，前 5 次调用全部返回 `errCode: -70002 / errMsg: "JSAPI_TRANSFER_FAILED" / data.err: {"err_msg":"system:access_denied"}`；**同一页面第二次加载后立刻全部成功**。原因：`WXU.API4` 里有 `h5AuthToken`/`needLogin` 等字段，登录鉴权令牌需要时间就绪。

**B 的对策（`callWithRetry`）**：`finderGetFollowList`（以及后续 `finderUserPage`）遇到 `errCode !== 0` 时，**指数退避重试**（如 2s→5s→10s，最多 4 次）；仍失败则在进度卡提示「视频号接口未就绪，请在微信里重新打开视频号页面后重试」。不要把首次 `-70002` 当成"没有关注"。

- 成功响应：`continueFlag:0, followCount:8, contactList.length:8`，每条 `username` 形如 `v2_...@finder`、`nickname`/`headUrl` 齐全。
