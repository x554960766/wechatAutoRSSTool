## 背景:为什么要做这件事

项目 `/Users/apple/Downloads/wechat-mp-tools` 是一个基于 **mitmproxy** 的视频号下载工具。代理已能正常 MITM `channels.weixin.qq.com`,页面也能打开。但当前三个功能坏掉:

1. **单视频下载按钮提示「未获取到秘钥」** —— 拿不到 `media.decodeKey`,下载的视频是加密的。
2. **作者页批量下载失败**。
3. **同步作者作品获取到 0 个视频**。

### 根因

当前注入脚本(`backend/mitm_proxy.py` 的 `get_injected_js_content()`)靠 **hook `fetch`/`XHR` 拦截明文 JSON 响应**来抓 feed。但视频号的 feed 接口返回的是**加密/protobuf**数据,`clone.json()` 解不出明文,更拿不到 `decodeKey`。所以三个功能全废。

参考工具 `ltaoo/wx_channels_download`(本仓 `wx_channels_download_temp/`,Go 实现)能正常工作,因为它走的是另一条路:**在 `res.wx.qq.com` 的 JS bundle 里 hook 视频号 App 自己的 `finderXxx` 方法**,等 App 内部把 feed 解密好之后,通过 `WXU.emit` 把含 `decodeKey` 的 feed 抛给注入脚本。

### 之前的失败尝试(必读,避免重蹈覆辙)

本会话曾尝试加回 bundle hook,结果**视频号页面白屏**。原因不是 hook 思路错,而是:

> bundle hook 注入的代码里调用了 `WXU.emit(...)`,但 `WXU` 这个全局对象**从未被定义**(它由参考工具一整套前置依赖脚本 `eventbus.js`+`utils.js` 提供)。我们只单独注入了 bundle hook、没注入这套依赖,导致 bundle 执行到 `WXU.emit` 时抛 `ReferenceError` → App 初始化崩溃 → 白屏。

**结论:bundle hook 不能单独存在,必须连同参考工具那一整套前端注入脚本(WXU 运行时)一起移植。** 这就是本计划要做的。

## 关键约束与架构差异(实现前必须理解)

### ⚠️ 差异 1:本项目是「后端下载」,参考工具是「前端下载」

参考工具的 `downloaderv2.js` 在**浏览器里**完成 fetch 视频流 → WASM 解密 → 保存,还要把下载任务发到伪装域名 `kf.qq.com`/`weixin110.qq.com`(反代到本地 HTTP 服务)+ WebSocket 推进度。

**本项目不需要这套。** 本项目的下载+ISAAC64 解密+落盘已经在 `backend/channels.py` 的 `/download` 里实现且可用(`channels.py:482`)。

**因此:不要移植** `downloaderv2.js`、`downloader.js`、伪装域名反代、`/api/task/*`、`/ws/downloader`、`FileSaver`/`jszip`/`recorder`/`axios` 这些前端下载器组件。

### ✅ 该移植的:只有「抓取层 + UI 层」

| 移植 | 不移植 |
|---|---|
| `eventbus.js`(WXE 事件总线) | `downloaderv2.js` / `downloader.js` |
| `utils.js`(WXU:feed 捕获、`format_feed`、`set_feed`) | 伪装域名反代(`kf.qq.com` 等) |
| bundle hook(发出 `Init`/`FeedProfileLoaded`/`UserFeedsLoaded` 事件) | `/api/task/*`、`/ws/downloader` |
| 页面脚本 `home.js`/`feed.js`/`profile.js` 里的**按钮 UI** | WASM 视频解密(`__wx_channels_decrypt`) |
| 第三方 UI 库(mitt、weui、wui、floating-ui、timeless) | `FileSaver`/`jszip`/`recorder`/`axios`/`getFeedInfo` |

### ✅ 该改接的:抓取层的「出口」改打本项目后端 API

参考工具脚本里所有「下载」「同步」动作,出口要从「参考工具的前端下载器」改成「本项目现有的后端 API」:

| 动作 | 本项目后端契约(已存在,不要改后端) |
|---|---|
| 下载单个视频 | `POST /__wx_channels_api/download`,body `{url, description, createtime, key}`(代理转发到 `127.0.0.1:5200/api/channels/download`)。`channels.py:482` |
| 同步作者作品 | `POST /__wx_channels_api/sync-feed`,body `{username, feeds:[<原始 feed>]}`(代理短路处理,`save_synced_feeds`)。**注意:feeds 是原始 feed 结构(含 `objectDesc.media[0].decodeKey`),不是 `format_feed` 后的对象。** `mitm_proxy.py:448` |
| 错误日志 | `POST /__wx_channels_api/log-error`(已存在) |

`/download` 的 `url` 字段构造规则(对齐 `save_synced_feeds`):`media.url + media.urlToken + "&X-snsvideoflag=" + spec.fileFormat`;`key` = `media.decodeKey`。

### ⚠️ 差异 2:正则引擎差异(Go RE2 vs Python re)

参考工具 bundle hook 正则如 `async finderInit\(\)\{(.*?)\}async`。Go 的 RE2 **默认 `.` 不匹配换行**。视频号 bundle 是单行压缩代码,所以 `.*?` 非贪婪能精确停在第一个 `}async`。

移植到 Python 时:
- bundle 是单行,**不要用 `re.S`/`re.DOTALL`**(本会话之前的死代码 `_inject_feed_capture_hooks` 用了 `re.S`,是隐患)。
- `(.*?)` 非贪婪行为与 Go 一致即可。
- 逐条核对每个正则的替换串与参考工具 `plugin.go:297-466` **完全一致**。

### ⚠️ 差异 3:wasm 文件必须跳过

`res.wx.qq.com` 上路径含 `wasm_video_decode` 的 JS **必须原样放行**,绝不能改写(`plugin.go:273`)。否则解密相关功能崩溃。

## bundle hook(`res.wx.qq.com` 上的 JS 改写)

参考 `plugin.go:266-526`。在 mitmproxy 的 `response` 钩子里,对 host=`res.wx.qq.com` 且 content-type 含 `javascript` 的响应处理:

1. **首先**:路径含 `wasm_video_decode` → 直接 return,不动。
2. 通用 import 改写(cache-busting):`from"X.js"` → `from"X.js?t=<ver>"` 等四条(`plugin.go:283-286`)。
3. 路径含 `virtual_svg-icons-register.publish`:
   - 对 `finderInit`/`finderGetCommentDetail`/`finderUserPage`/`finderPcFlow`/`finderGetRecommend` 等方法做包裹,执行后 `WXU.emit(WXU.Events.XXX, result.data...)`。逐条对照 `plugin.go:299-436` 的替换串。**最关键的三个**:
     - `finderInit` → `WXU.emit(WXU.Events.Init, result.data)`(拿 `mainFinderUsername`)
     - `finderGetCommentDetail` → `WXU.emit(WXU.Events.FeedProfileLoaded, result.data.object)`(**单视频 decodeKey 来源**)
     - `finderUserPage` → `WXU.emit(WXU.Events.UserFeedsLoaded, result.data.object)`(作者页列表)
   - export hook:把 `export{` 替成 `;WXU.emit(WXU.Events.APILoaded,{...});export{`(`plugin.go:437-464`),让 `WXU.API.finderUserPage` 可被 profile.js 主动调用。
4. 路径含 `connect.publish`/`applyMic.publish`:滚动流 hook(`plugin.go:468-522`)。**第一版可暂缓**,先保证 home/feed/profile 三个核心场景。

### Python 移植注意
- 不用 `re.DOTALL`。
- 提取流变量名用 `flowTab:([a-zA-Z_$]+)`,fallback `"yt"`;`localFlowTab:` fallback `"vn"`。
- 每条替换后可加日志 `[Bundle Hook] hooked finderXxx`,便于验证命中。

## 调试锦囊

- **白屏**:必是 bundle hook 改坏了 bundle,或注入顺序错导致 `WXU`/`WUI`/`Timeless` 未定义。先在浏览器/webview 控制台看第一条报错。本项目注入脚本已有 `/__wx_channels_api/log-error` 上报通道,看后端 `[FRONTEND ERROR]` 日志。
- **按钮出现但没数据**:bundle hook 没命中(看 `[Bundle Hook]` 日志);或 `res.wx.qq.com` 没被 MITM(看 `allow_hosts`);或微信用了缓存的旧 bundle(看 cache-busting 是否生效,必要时清微信缓存 `/api/channels/clear-cache`)。
- **decodeKey 为空**:`FeedProfileLoaded` 事件没发出或 `media.decodeKey` 路径变了。打印 hook 后 `result.data.object` 的结构核对。
- **逐条二分定位坏正则**:先只开 import 改写(不开 finder hook)确认不白屏,再逐个加 finder hook,每加一个测一次。

## ⚠️ 已知待验证隐患(实现/真机阶段重点检查)

> 截至本计划更新时,代码已按步骤 1-5 实现(资源已复制、HTML 注入、bundle hook、出口对接均到位)。以下两点是 review 时发现的、**尚未真机验证**的隐患,务必在测试时重点确认:

1. **`export{` 多次替换问题**:`_hook_wx_bundle` 里 `re.sub(r'exports?\s*\{', js_wxapi, js_script)`(`virtual_svg-icons-register.publish` 分支约 811 行、`connect.publish` 分支约 839 行)**没有 `count=1`**。参考工具(Go `ReplaceAllString` 在该处实际只命中一次)的 bundle 通常只有一个 `export{`,但若当前微信 bundle 含多个 `export{`,会注入多段 `WXU.emit(...);export{` → 可能语法错误/重复 emit → **白屏或重复触发**。若白屏,优先在这两处加 `count=1` 复测。

2. **sync-feed 契约不匹配(很可能是「同步仍为 0」的原因)**:`utils.js` 的 `create_batch`(约 842 行)发给 `/__wx_channels_api/sync-feed` 的 `feeds` 是 **`format_feed` 后的扁平对象**(字段为 `feed.url`、`feed.key`、`feed.contact.id`、`feed.title`)。但后端 `save_synced_feeds`(`mitm_proxy.py:530`)期望 **原始 feed 结构**,读的是 `feed.objectDesc.media[0].decodeKey`、`feed.objectDesc.mediaType`、`feed.objectDesc.description`、`feed.contact.username`。**两者字段路径对不上 → 后端 `media_type != 4` 判断失败 → 整批跳过 → 同步落盘 0 条。**
   - **修法二选一**:(a) 改 `save_synced_feeds` 兼容扁平结构(检测 `feed.get("type")=="media"` 或直接读 `feed["url"]/feed["key"]`);或 (b) 让前端 `create_batch` 传**原始 feed**(profile.js 拿到的 `r.data.object` 原始项,而非 `format_feed` 后的)给 sync-feed,只在 `download` 出口用 format 后的字段。**推荐 (a)**,改动集中在后端一处,不碰前端脚本。
   - 验证:同步后看后端日志 `[FRONTEND PROFILE]`/`save_synced_feeds` 是否报错,以及 `channels_parsed_feeds.json` 是否真的写入了带 `decode_key` 的条目。
